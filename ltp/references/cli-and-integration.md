# CLI, generated tree, dashboard, and integration

## Install

```bash
pip install -e ltp        # editable install of the engine + `ltp` console script
```

The engine is pure standard library plus PyYAML (the one parsing dependency).

## The generated tree

Authored (one file):

```
<project>/ltp/ltp-model.yaml
```

Generated by `ltp sync` (never hand-edited):

```
<project>/ltp/generated/
  00-coverage-and-evidence.md
  01-project-model.md
  02-goal-tree.md
  03-current-reality-tree.md
  04-evaporating-clouds.md
  05-future-reality-tree.md
  06-prerequisite-tree.md
  07-transition-tree.md
  08-next-action.md
  09-assumptions-and-questions.md
  diagrams/<view>.mmd            # one per non-empty view
  dashboard-model.json           # normalized data for the dashboard
```

Only non-empty views get a diagram, so a `diagnose`-mode analysis produces a
Goal-Tree and Current-Reality diagram and nothing spurious. Each Markdown/Mermaid
file carries a "generated -- do not edit" header; `dashboard-model.json` carries a
`build.source_hash` (a stable hash of the whole model) and generator version.

## CLI reference

| Command | Writes? | Exit codes | Purpose |
| --- | --- | --- | --- |
| `ltp init [--name N] [--force]` | model | 0 / 2 exists | scaffold a valid starter |
| `ltp validate [--strict] [--json]` | no | 0 ok / 1 errors (or warnings if `--strict`) | logical diagnostics |
| `ltp render` | generated | 0 | write projections, no publish gate |
| `ltp sync [--force]` | generated | 0 / 2 refused | validate then write |
| `ltp check` | no | 0 / 2 stale-or-invalid | CI gate |
| `ltp doctor` | no | 0 / 1 errors | diagnostics + staleness notes |
| `ltp migrate [--write]` | model | 0 / 2 | v1 -> v2, preserving ids |
| `ltp explain ID` | no | 0 / 2 not-found | trace one record |

`--root` points at the project (default cwd; the engine looks for `ltp/ltp-model.yaml`);
`--model` points directly at a model file.

## CI

```bash
python -m pytest                                                # engine tests
ltp --model ltp/ltp-model.yaml validate --strict               # logic must hold
ltp --root . check                                             # projections must be current
npm ci --prefix ltp/dashboard && npm run build --prefix ltp/dashboard   # dashboard builds
```

`node_modules/` and `ltp/dashboard/dist/` are git-ignored; the dashboard builds
from `package-lock.json` with `npm ci`.

## Dashboard

The dashboard is a read-only view of `generated/dashboard-model.json` -- validated,
derived, with compound sufficiency already expanded into explicit gate nodes and
each view's graph precomputed. It never parses the raw YAML.

```bash
ltp --root /path/to/project sync            # produce the generated data first
python ltp/scripts/serve_dashboard.py --project /path/to/project --open
```

The server binds to `127.0.0.1` and serves only the bundled frontend and the
generated JSON. Use `--host` with `--allow-network` only when the user explicitly
asks for network exposure. The dashboard's model-health panel shows the same
diagnostics `ltp validate` prints, so it reveals logical incompleteness rather than
merely drawing the model attractively.

## Integration with `hypothesize`

`ltp` owns the causal structure and its logical scrutiny; `hypothesize` owns the
empirical status of tested claims. The bridge is thin -- keep the implementations
separate.

Link an LTP claim to a hypothesis, recording what the test can establish:

```yaml
causal_claims:
  - id: CLM-17
    verification:
      hypothesis_ref: HYP-T2-REALISM
      role: causal_outcome     # entity_existence | mechanism | causal_outcome |
                               # scientific_outcome | negative_branch_guardrail |
                               # prerequisite_readiness | transition_acceptance
```

Keep logical and empirical status distinct. A green implementation test may prove
an injection *exists* (`entity_existence`); it does not prove the injection
*causes* the modelled effect. A falsified empirical hypothesis should mark the
linked claim's `review_status: disputed` and its dependents for review -- it must
**not** silently delete or reverse the claim.

Bridge commands (see `ltp hypotheses --help`):

```bash
ltp hypotheses export     # list claims that carry a verification link
ltp hypotheses check      # report links whose hypothesis_ref cannot be resolved
ltp evidence import PATH  # fold a hypothesize research-status.json into review_status
```

| LTP element | `hypothesize` role |
| --- | --- |
| entity-existence reservation | observation hypothesis |
| CRT/FRT causal claim | causal hypothesis |
| injection implementation | capability / mechanism test |
| FRT predicted effect | outcome hypothesis |
| negative branch | guardrail hypothesis |
| PRT prerequisite | readiness hypothesis |
| TT immediate effect | acceptance test |
